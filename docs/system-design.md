# 云汀智能家居小程序系统设计

## 1. 设计目标

云汀智能家居小程序面向普通用户提供设备绑定、设备管理和自动浇水系统控制能力。系统需要解决三个核心问题：

- 如何判断用户已经登录。
- 用户通过手机号验证码登录成功后，下次打开是否还需要再次登录。
- 云端服务器如何承载登录、设备、浇水配置、设备指令和设备状态等业务。

本设计以微信小程序 + 云开发为基础。小程序端负责交互和本地会话缓存，云端负责身份校验、权限判断、数据持久化和设备通信。任何涉及用户身份、设备归属、控制指令的最终判断都应在云端完成。

登录模块的详细规范见 [login-module-design.md](login-module-design.md)。后续手机端和云端实现登录、验证码、会话、账号复用时，应以该文档为准。

## 2. 登录态定义

### 2.1 用户身份

系统中用户身份由两部分组成：

- 微信身份：云函数通过 `cloud.getWXContext()` 获取 `OPENID`，用于识别当前微信小程序用户。
- 手机号身份：用户通过短信验证码证明自己拥有该手机号，用于业务账号、设备绑定和售后联系。

推荐账号模型为：一个 `user` 可以绑定一个主手机号和一个或多个微信 `openid`。MVP 阶段可以先采用一个手机号对应一个用户、一个微信 `openid` 绑定一个用户的简单模型。

### 2.2 已登录的判断标准

客户端不能只靠本地缓存判断用户已登录。本地缓存只能作为快速进入应用的依据，最终以云端会话校验结果为准。

用户被认为“已登录”需要同时满足：

- 小程序本地存在 `authSession`，包含 `sessionId`、`userId`、手机号脱敏信息、`expiresAt`。
- 本地 `expiresAt` 晚于当前时间。
- 小程序启动或进入需要鉴权的页面时，调用云函数 `auth.checkSession`。
- 云端确认该 `sessionId` 存在、未过期、未注销、未被风控冻结。
- 云端确认该 `sessionId` 归属当前微信 `OPENID` 和对应 `userId`。
- 用户状态为正常，手机号绑定状态有效。

结论：客户端可以先根据本地会话做页面跳转，但必须尽快请求云端确认。云端确认失败时，客户端清除本地会话并回到登录页。

### 2.3 下次是否需要再次登录

用户已经收到验证码并登录成功后，下次是否需要再次登录取决于云端会话是否仍然有效。

推荐策略：

- 验证码只用于本次登录校验，成功后立即标记为已使用，不作为长期登录凭据。
- 登录成功后云端生成 `sessionId`，有效期建议 30 天。
- 每次打开小程序或调用核心接口时，云端可以刷新会话有效期，形成滑动过期机制。
- 可以设置最长会话生命周期，例如 90 天。超过最长生命周期即使中间持续使用，也要求重新短信登录。
- 用户主动退出登录、手机号解绑、账号冻结、检测到风险登录时，会话立即失效。

因此：同一微信账号、同一小程序本地缓存未被清除、会话仍有效时，下次不需要再次输入验证码。会话过期、本地缓存被清除、换微信账号、换设备或云端风控要求重新认证时，需要重新登录。

## 3. 端到端登录流程

### 3.1 首次登录

1. 用户输入手机号。
2. 小程序调用 `auth.sendCode`。
3. 云端校验手机号格式、发送频率、IP/OPENID 风控限制。
4. 云端生成 6 位验证码，只保存验证码哈希值、过期时间和尝试次数。
5. 云端调用短信服务商发送验证码。
6. 用户输入验证码。
7. 小程序调用 `auth.loginByCode`，传入手机号和验证码。
8. 云端校验验证码是否正确、未过期、未使用、未超过尝试次数。
9. 云端根据手机号和 `OPENID` 查找或创建用户。
10. 云端创建 `session` 记录并返回 `sessionId`、`userId`、脱敏手机号、过期时间。
11. 小程序将会话保存到 `wx.setStorageSync('authSession')`。
12. 小程序跳转设备管理页。

### 3.2 再次打开小程序

1. 小程序读取本地 `authSession`。
2. 本地不存在或已过期，直接进入登录页。
3. 本地存在且未过期，先进入设备管理页或显示启动态。
4. 小程序调用 `auth.checkSession`。
5. 云端校验会话和当前 `OPENID`。
6. 校验成功则返回最新用户信息和新的过期时间。
7. 小程序更新本地 `authSession`。
8. 校验失败则清除本地会话并跳转登录页。

### 3.3 退出登录

1. 用户点击退出。
2. 小程序调用 `auth.logout`。
3. 云端将当前 `session` 标记为 `revoked`。
4. 小程序清除本地 `authSession`。
5. 小程序跳转登录页。

## 4. 云端总体架构

### 4.1 推荐分层

云端建议按以下职责拆分：

- 云函数 API 层：接收小程序请求，校验参数，统一返回格式。
- 鉴权中间层：解析 `sessionId`，校验 `OPENID`、用户状态、会话状态。
- 业务服务层：处理用户、设备、浇水配置、设备指令等业务规则。
- 数据访问层：封装数据库集合读写。
- 外部服务适配层：短信服务、IoT 平台、消息推送、日志告警。

小程序不直接写核心业务数据库。所有设备归属、配置保存、指令下发都通过云函数完成。

### 4.2 云函数模块

推荐云函数按业务拆分，MVP 阶段也可以先放在一个 `api` 云函数中用 `type` 路由。

| 模块 | 云函数 | 说明 |
| --- | --- | --- |
| 鉴权 | `auth.sendCode` | 发送短信验证码 |
| 鉴权 | `auth.loginByCode` | 验证码登录并创建会话 |
| 鉴权 | `auth.checkSession` | 校验并刷新会话 |
| 鉴权 | `auth.logout` | 注销当前会话 |
| 用户 | `user.getProfile` | 获取当前用户资料 |
| 设备 | `device.bind` | 绑定设备号 |
| 设备 | `device.list` | 查询当前用户设备列表 |
| 设备 | `device.get` | 查询设备详情 |
| 设备 | `device.unbind` | 解绑设备 |
| 浇水 | `watering.getConfig` | 获取浇水配置 |
| 浇水 | `watering.saveConfig` | 保存浇水模式和参数 |
| 浇水 | `watering.startManual` | 手动开始浇水 |
| 浇水 | `watering.stopManual` | 手动停止浇水 |
| 设备通信 | `deviceCommand.dispatch` | 生成并下发设备指令 |
| 设备通信 | `deviceTelemetry.report` | 接收设备状态和传感器数据 |

### 4.3 统一接口返回

所有云函数建议统一返回：

```json
{
  "success": true,
  "code": "OK",
  "message": "",
  "data": {}
}
```

错误返回示例：

```json
{
  "success": false,
  "code": "SESSION_EXPIRED",
  "message": "登录已过期，请重新登录",
  "data": null
}
```

客户端根据 `code` 做统一处理。遇到 `SESSION_EXPIRED`、`SESSION_REVOKED`、`USER_DISABLED` 时清除本地会话并跳转登录页。

## 5. 数据模型设计

### 5.1 `users` 用户集合

```json
{
  "_id": "user_xxx",
  "phone": "13800138000",
  "phoneMasked": "138****8000",
  "openids": ["openid_xxx"],
  "status": "active",
  "createdAt": 1710000000000,
  "updatedAt": 1710000000000,
  "lastLoginAt": 1710000000000
}
```

业务规则：手机号唯一。`status` 为 `disabled` 时禁止登录和操作设备。

### 5.2 `sms_codes` 验证码集合

```json
{
  "_id": "code_xxx",
  "phone": "13800138000",
  "openid": "openid_xxx",
  "codeHash": "sha256_hash",
  "scene": "login",
  "status": "pending",
  "expiresAt": 1710000300000,
  "attempts": 0,
  "createdAt": 1710000000000,
  "usedAt": null
}
```

业务规则：验证码有效期建议 5 分钟；同一手机号 60 秒内只能发送一次；同一手机号每天发送次数需要限制；验证码校验失败次数超过 5 次后作废。

### 5.3 `sessions` 会话集合

```json
{
  "_id": "session_xxx",
  "userId": "user_xxx",
  "openid": "openid_xxx",
  "status": "active",
  "createdAt": 1710000000000,
  "expiresAt": 1712592000000,
  "maxExpiresAt": 1717776000000,
  "lastSeenAt": 1710000000000,
  "revokedAt": null
}
```

业务规则：会话必须绑定 `openid`。任何接口传入的 `sessionId` 都必须和当前云函数上下文中的 `OPENID` 匹配。

### 5.4 `devices` 设备集合

```json
{
  "_id": "device_xxx",
  "deviceNo": "YT-WATER-001",
  "deviceSecretHash": "hash",
  "type": "watering",
  "name": "阳台自动浇水",
  "ownerUserId": "user_xxx",
  "status": "online",
  "bindStatus": "bound",
  "boundAt": 1710000000000,
  "lastOnlineAt": 1710000000000,
  "createdAt": 1710000000000,
  "updatedAt": 1710000000000
}
```

业务规则：设备号唯一。绑定设备时必须校验设备号存在、未被其他用户绑定、设备密钥或出厂绑定码正确。

### 5.5 `watering_configs` 浇水配置集合

```json
{
  "_id": "config_xxx",
  "deviceId": "device_xxx",
  "ownerUserId": "user_xxx",
  "mode": "demand",
  "demand": {
    "intervalHours": 4,
    "threshold": 35,
    "durationSeconds": 20
  },
  "schedule": {
    "timesPerDay": 2,
    "durationSeconds": 30
  },
  "manual": {
    "durationSeconds": 10
  },
  "version": 1,
  "updatedAt": 1710000000000
}
```

业务规则：保存配置前必须校验当前用户拥有该设备。配置保存成功后生成一条设备配置同步指令。

### 5.6 `device_commands` 设备指令集合

```json
{
  "_id": "cmd_xxx",
  "deviceId": "device_xxx",
  "ownerUserId": "user_xxx",
  "type": "watering.startManual",
  "payload": {
    "durationSeconds": 10
  },
  "status": "pending",
  "createdAt": 1710000000000,
  "sentAt": null,
  "ackAt": null,
  "failedReason": ""
}
```

业务规则：所有控制设备的动作都先生成指令记录，再由云端下发到 IoT 平台或设备通信服务。指令需要有超时、重试和失败状态。

### 5.7 `device_telemetry` 设备上报集合

```json
{
  "_id": "telemetry_xxx",
  "deviceId": "device_xxx",
  "type": "soil_moisture",
  "value": 32,
  "unit": "%",
  "reportedAt": 1710000000000
}
```

业务规则：设备上报的数据用于展示设备状态、判断在线状态、生成告警和辅助按需浇水策略。

## 6. 功能模块定义

### 6.1 小程序启动与路由守卫

功能职责：判断用户进入登录页还是设备管理页。

业务逻辑：

- `app.onLaunch` 初始化云开发环境。
- 读取本地 `authSession`。
- 无会话或本地会话过期，进入登录页。
- 有会话则进入设备管理页并调用 `auth.checkSession`。
- 云端校验失败，清除本地会话并回登录页。

### 6.2 手机号验证码登录模块

功能职责：完成手机号验证码发送、验证码校验和会话创建。

业务逻辑：

- 手机号必须符合大陆手机号格式。
- 验证码由云端生成，不能由前端生成。
- 验证码只保存哈希，不保存明文。
- 验证码成功使用后立即置为 `used`。
- 登录成功后创建用户或更新用户最后登录时间。
- 登录成功后创建会话并返回给小程序。

### 6.3 用户模块

功能职责：维护用户基础信息和账号状态。

业务逻辑：

- 用户首次手机号登录时创建用户。
- 同一手机号再次登录时复用已有用户。
- 如果同一微信 `OPENID` 尝试绑定不同手机号，需要根据产品策略决定是否允许换绑。
- 用户被禁用后，所有会话校验失败。

### 6.4 设备绑定模块

功能职责：让用户把设备号绑定到自己的账号下。

业务逻辑：

- 用户必须登录。
- 设备号不能为空，并符合设备号格式。
- 云端检查设备是否存在。
- 云端检查设备是否已经绑定其他用户。
- 推荐要求用户输入设备出厂绑定码或扫描设备二维码，防止仅凭设备号被误绑。
- 绑定成功后写入 `devices.ownerUserId`，创建设备默认配置。

### 6.5 设备列表模块

功能职责：展示当前用户下所有设备并支持按类型筛选。

业务逻辑：

- 用户必须登录。
- 只查询 `ownerUserId` 等于当前用户的设备。
- 返回设备名称、类型、在线状态、设备号脱敏或完整设备号、最近在线时间。
- 设备类型包括自动浇水系统、环境传感器、智能灯控、智能插座等。

### 6.6 自动浇水配置模块

功能职责：管理自动浇水系统的三种浇水模式。

业务逻辑：

- 用户必须拥有该设备。
- 模式只能是 `demand`、`schedule`、`manual` 之一。
- 按需浇水参数：检测周期 1-72 小时，湿度阈值 1-100%，浇水时长 1-3600 秒。
- 定期浇水参数：每天浇水次数 1-24 次，每次浇水时长 1-3600 秒。
- 手动浇水参数：浇水时长 1-3600 秒。
- 保存配置成功后，云端需要生成配置同步指令，让设备应用最新策略。

### 6.7 手动浇水控制模块

功能职责：用户点击开始后立即触发设备浇水。

业务逻辑：

- 用户必须拥有该设备。
- 设备必须在线，离线时返回不可操作。
- 云端生成 `watering.startManual` 指令。
- 指令下发成功后返回执行中状态。
- 设备确认后更新指令状态和设备最近浇水时间。
- 用户点击停止时生成 `watering.stopManual` 指令。
- 如果设备未确认，客户端应展示“指令已发送”或“等待设备响应”，不能假定设备已经执行。

### 6.8 设备状态与传感器数据模块

功能职责：接收设备在线状态、土壤湿度、电量、水箱状态等数据。

业务逻辑：

- 设备上报必须使用设备密钥或 IoT 平台签名认证。
- 云端根据最近上报时间判断在线状态。
- 土壤湿度数据可用于前端展示，也可用于云端告警。
- MVP 阶段按需浇水策略可以由设备本地执行；云端负责下发配置和保存状态。

## 7. 安全与权限规则

- 所有核心接口必须校验 `sessionId`。
- 云端必须使用 `cloud.getWXContext().OPENID` 与会话记录匹配，不能信任前端传入的 `openid`。
- 前端不能直接更新 `devices`、`watering_configs`、`device_commands` 等核心集合。
- 设备控制接口必须校验设备归属。
- 验证码接口必须限制发送频率和校验次数。
- 设备绑定必须使用设备号 + 绑定码、二维码签名或出厂密钥，不能只靠设备号。
- 会话失效、账号冻结、设备解绑后，相关操作应立即失败。

## 8. 当前原型到云端版本的迁移路径

当前小程序已经用本地缓存跑通了登录、设备绑定和浇水配置交互。迁移到云端建议分三步：

1. 接入真实鉴权：新增 `auth.sendCode`、`auth.loginByCode`、`auth.checkSession`、`auth.logout`，替换前端演示验证码和本地登录态。
2. 接入云数据库：新增 `users`、`sessions`、`devices`、`watering_configs` 集合，把设备列表和浇水配置从本地缓存迁移到云端。
3. 接入设备通信：新增 `device_commands`、`device_telemetry`，将保存配置、开始浇水、停止浇水对接 IoT 平台或设备通信服务。

## 9. 推荐默认策略

- 登录会话有效期：30 天。
- 最长会话生命周期：90 天。
- 验证码有效期：5 分钟。
- 验证码重发间隔：60 秒。
- 单手机号每日验证码上限：10 次。
- 单验证码最大错误次数：5 次。
- 手动浇水时长范围：1-3600 秒。
- 设备离线判断：超过 5 分钟未上报心跳则视为离线。

这些值可以先作为 MVP 默认配置，后续根据真实设备和用户使用情况调整。
